1""" 

2State Processors Module 

3 

4Modular state processors for S1→SN configurator flow. 

5 

6Each state in the configurator has a dedicated processor that handles: 

7- Product search via SearchOrchestrator 

8- State transition logic (get_next_state) 

9- Multi-select behavior (for accessories only) 

10- Proactive preview generation 

11- Zero-results handling 

12 

13Architecture: 

14 StateProcessor (base.py) 

15 ├── PowerSourceStateProcessor (power_source.py) 

16 ├── FeederStateProcessor (feeder.py) 

17 ├── CoolerStateProcessor (cooler.py) 

18 ├── InterconnectorStateProcessor (interconnector.py) 

19 ├── TorchStateProcessor (torch.py) 

20 └── AccessoryStateProcessor (accessory.py) 

21 └── 9 accessory states via factory 

22 

23Registry: 

24 StateProcessorRegistry (registry.py) 

25 - Central registry for all 14 processors 

26 - Dependency injection (SearchOrchestrator) 

27 - Configuration loading (state_config.json) 

28 - Processor retrieval by state name 

29 

30Usage: 

31 # Initialize registry 

32 from app.services.processors import init_state_processor_registry, get_state_processor_registry 

33 

34 registry = init_state_processor_registry( 

35 state_config_path="app/config/state_config.json", 

36 search_orchestrator=search_orchestrator_instance 

37 ) 

38 

39 # Get processor for current state 

40 processor = registry.get_processor("power_source_selection") 

41 

42 # Execute search 

43 results = await processor.search_products( 

44 user_message="I need a 500A MIG welder", 

45 master_parameters={...}, 

46 selected_components={}, 

47 limit=10, 

48 offset=0 

49 ) 

50 

51 # Get next state 

52 next_state = processor.get_next_state( 

53 conversation_state=state, 

54 selection_made=True 

55 )
