    Infrastructure Sizing¶

    Target Load Profile¶

    Baseline Requirements: - Concurrent Users: 10 active users - PostgreSQL Data: ~1 GB (session archives, logs, analytics) - Neo4j Data: 2-3 GB (product catalog, compatibility graph) - Expected Growth: 20% annually - Geographic Distribution: Single region deployment - Availability SLA: 99.5% (43.8 hours downtime/year)

    Performance Targets¶

    Metric Target Measurement
    API Response Time (p50) < 200ms Median response time
    API Response Time (p95) < 500ms 95th percentile
    API Response Time (p99) < 1000ms 99th percentile
    Concurrent Sessions 10-15 Peak concurrent users
    Session Duration 5-15 min Average session length
    Database Query Time < 100ms Neo4j compatibility queries
    LLM Response Time < 2000ms OpenAI API calls
    Uptime 99.5% Monthly availability

    conversation_state JSONB NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

);

-- Indexes for performance
CREATE INDEX idx_archived_sessions_user_id ON archived_sessions(user_id);

CREATE INDEX idx_archived_sessions_session_id ON archived_sessions(session_id);

-- GIN index for JSONB queries (if needed)
CREATE INDEX idx_archived_sessions_state ON archived_sessions USING GIN (conversation_state);

    Graph Schema¶

    Node Labels: - PowerSource (~50 nodes) - Feeder (~40 nodes) - Cooler (~30 nodes) - Interconnector (~25 nodes) - Torch (~60 nodes) - Accessory (~500 nodes)

    Total Nodes: ~705 nodes

    Relationships: - COMPATIBLE_WITH (~5,000 relationships) - Average degree: ~14 relationships per node

    Estimated Graph Size: 2.5 GB (nodes + relationships + indexes)

CREATE CONSTRAINT power_source_gin IF NOT EXISTS
FOR (ps:PowerSource) REQUIRE ps.gin IS UNIQUE;

CREATE CONSTRAINT feeder_gin IF NOT EXISTS
FOR (f:Feeder) REQUIRE f.gin IS UNIQUE;

CREATE CONSTRAINT cooler_gin IF NOT EXISTS
FOR (c:Cooler) REQUIRE c.gin IS UNIQUE;

CREATE CONSTRAINT interconnector_gin IF NOT EXISTS
FOR (i:Interconnector) REQUIRE i.gin IS UNIQUE;

CREATE CONSTRAINT torch_gin IF NOT EXISTS
FOR (t:Torch) REQUIRE t.gin IS UNIQUE;

CREATE CONSTRAINT accessory_gin IF NOT EXISTS
FOR (a:Accessory) REQUIRE a.gin IS UNIQUE;

-- Composite indexes for search performance
CREATE INDEX power_source_search IF NOT EXISTS
FOR (ps:PowerSource) ON (ps.category, ps.current_output);

CREATE INDEX feeder_search IF NOT EXISTS
FOR (f:Feeder) ON (f.category, f.cooling_type);

CREATE INDEX accessory_search IF NOT EXISTS
FOR (a:Accessory) ON (a.category, a.accessory_category);

-- Full-text search indexes
CALL db.index.fulltext.createNodeIndex(
  'productNameSearch',
  ['PowerSource', 'Feeder', 'Cooler', 'Interconnector', 'Torch', 'Accessory'],
  ['item_name', 'description_catalogue']
);

maxmemory 256mb
maxmemory-policy allkeys-lru        # Evict least recently used keys

# Persistence (for session safety)
appendonly yes

# Logging

logfile "/var/log/redis/redis-server.log"

# Performance
slowlog-log-slower-than 10000       # Log queries > 10ms
slowlog-max-len 128

    Server Specifications¶

    Application Server (Single-Server Deployment)¶

    Recommended Configuration for 10 Concurrent Users:

    Component Specification Justification
    Operating System Ubuntu 22.04 LTS Long-term support, stability
    vCPUs 4 cores 4 Uvicorn workers + background tasks
    RAM 8 GB Backend (4GB) + PostgreSQL (1GB) + Redis (256MB) + OS (2GB) + Buffer (1GB)
    Storage 50 GB SSD OS (10GB) + Logs (5GB) + PostgreSQL (10GB) + Backups (20GB) + Buffer (5GB)
    Network 1 Gbps Sufficient for 10 concurrent users
    IOPS 3000 General Purpose SSD performance

    Cloud Provider Equivalents¶

    AWS EC2: - Instance Type: t3.medium or t3a.medium - vCPUs: 2 vCPUs (4 threads with hyperthreading) - RAM: 4 GB (upgrade to t3.large with 8GB for production) - Storage: 50 GB gp3 SSD (3000 IOPS) - Estimated Cost: $30-50/month

    Recommended: t3.large (2 vCPU, 8GB RAM) for production stability

    Azure Virtual Machines: - Instance Type: Standard_B2s or Standard_D2s_v3 - vCPUs: 2 vCPUs - RAM: 4-8 GB (choose D2s_v3 with 8GB) - Storage: 50 GB Premium SSD (P10) - Estimated Cost: $40-60/month

    Recommended: Standard_D2s_v3 (2 vCPU, 8GB RAM)

    Google Cloud Compute Engine: - Instance Type: e2-standard-2 - vCPUs: 2 vCPUs - RAM: 8 GB - Storage: 50 GB SSD persistent disk - Estimated Cost: $35-55/month

    On-Premises Server¶

    Minimum Specifications: - CPU: Intel Xeon E3 or AMD Ryzen 5 (4 cores, 3.0+ GHz) - RAM: 8 GB DDR4 - Storage: 250 GB SSD (or 2× 500 GB in RAID 1) - Network: 1 Gbps Ethernet - Power: Redundant PSU (optional for HA)

    Neo4j Server (Separate Server Recommended)¶

    Option 1: Neo4j Aura (Managed Cloud - Recommended)

    Tier Specification Cost Use Case
    Aura Professional 2GB RAM, 8GB Storage ~$65/month Production (10 users)
    Aura Enterprise 4GB RAM, 16GB Storage ~$200/month High Availability (25+ users)

    Advantages: - ✅ Fully managed (automatic backups, updates, scaling) - ✅ Built-in monitoring and alerting - ✅ 99.95% SLA (Enterprise tier) - ✅ No server maintenance required

    Option 2: Self-Hosted Neo4j Community Edition

    Component Specification Justification
    Operating System Ubuntu 22.04 LTS Recommended by Neo4j
    vCPUs 2 cores Graph database queries
    RAM 4 GB 1GB heap + 1GB page cache + 2GB OS
    Storage 20 GB SSD 3GB data + 5GB indexes + 10GB growth + 2GB logs
    Java OpenJDK 17 Required for Neo4j 5.x

    Cloud Provider Equivalents: - AWS: t3.medium (2 vCPU, 4GB RAM) - $30/month - Azure: Standard_B2s (2 vCPU, 4GB RAM) - $35/month - GCP: e2-medium (2 vCPU, 4GB RAM) - $25/month

    Note: Neo4j Aura is recommended for production to avoid operational overhead.

# SSH (admin only - replace with your IP)
sudo ufw allow from <ADMIN_IP> to any port 22

sudo ufw deny 5432/tcp comment 'PostgreSQL - localhost only'

# Neo4j (if self-hosted on same server)
sudo ufw deny 7474/tcp comment 'Neo4j HTTP - localhost only'
sudo ufw deny 7687/tcp comment 'Neo4j Bolt - localhost only'

# Allow outbound HTTPS
sudo ufw allow out 443/tcp comment 'HTTPS outbound'

# Check status
sudo ufw status verbose

    #!/bin/bash
# health-check.sh - Monitor application health

API_URL="http://localhost:8000/health"
ALERT_EMAIL="admin@example.com"

# Check health endpoint
response=$(curl -s -o /dev/null -w "%{http_code}" $API_URL)

if [ "$response" != "200" ]; then
    echo "ALERT: Health check failed (HTTP $response)" | \
        mail -s "ESAB Recommender Health Alert" $ALERT_EMAIL
    exit 1
fi

# Check individual components
health_json=$(curl -s $API_URL)
neo4j_status=$(echo $health_json | jq -r '.components.neo4j.status')
postgres_status=$(echo $health_json | jq -r '.components.postgres.status')
redis_status=$(echo $health_json | jq -r '.components.redis.status')

if [ "$neo4j_status" != "healthy" ]; then
    echo "ALERT: Neo4j unhealthy" | mail -s "Neo4j Alert" $ALERT_EMAIL
fi

if [ "$postgres_status" != "healthy" ]; then
    echo "ALERT: PostgreSQL unhealthy" | mail -s "PostgreSQL Alert" $ALERT_EMAIL
fi

if [ "$redis_status" != "healthy" ]; then
    echo "ALERT: Redis unhealthy" | mail -s "Redis Alert" $ALERT_EMAIL
fi

exit 0

    #!/bin/bash
# /opt/esab-recommender/scripts/backup-postgres.sh

BACKUP_DIR="/backup/postgresql"
RETENTION_DAYS=7
DB_NAME="pconfig"
DB_USER="postgres"

# Create backup directory
mkdir -p $BACKUP_DIR

# Generate backup filename
BACKUP_FILE="$BACKUP_DIR/pconfig_$(date +%Y%m%d_%H%M%S).dump"

# Create backup
pg_dump -U $DB_USER -Fc -f $BACKUP_FILE $DB_NAME

# Verify backup
if [ $? -eq 0 ]; then
    echo "Backup successful: $BACKUP_FILE"

    # Delete old backups (older than retention period)
    find $BACKUP_DIR -name "pconfig_*.dump" -mtime +$RETENTION_DAYS -delete

    # Log backup size
    du -h $BACKUP_FILE
else
    echo "Backup failed!" | mail -s "PostgreSQL Backup Failed" admin@example.com
    exit 1
fi

    Recovery Time Objective (RTO): 4 hours Recovery Point Objective (RPO): 24 hours (daily backups)

    1. Provision new server (30 minutes)
    2. Launch VM with same specifications

    3. Install operating system and dependencies

    4. Restore application code (15 minutes)

    5. Clone Git repository
    6. Install Python dependencies

    7. Configure environment variables

    8. Restore PostgreSQL (1 hour)

    9. Install PostgreSQL
    10. Restore latest backup

    11. Verify data integrity

    12. Restore Neo4j (2 hours)

    13. Option A: Connect to Neo4j Aura (no restore needed)

    15. Configure and start services (30 minutes)
