    Test Types¶

    Unit Tests (tests/unit/)¶

    Purpose: Test individual functions/classes in complete isolation

    Characteristics: - ‚ö° Fast: < 100ms per test - üîí Isolated: All external dependencies mocked - üéØ Focused: One function/class per test file - üîÑ Reliable: No flaky behavior, deterministic

    What to test: - Business logic - Data transformations - Edge cases and error handling - Input validation - Model serialization/deserialization

    What NOT to test: - External API integrations - Database operations - Network calls - File system operations

import pytest
from app.models.conversation import ConversationState, ConfiguratorState

@pytest.mark.unit
def test_conversation_state_default_initialization():
    """Test ConversationState initializes with correct defaults"""
    state = ConversationState(
        session_id="test-123",
        current_state=ConfiguratorState.POWER_SOURCE_SELECTION
    )

    assert state.session_id == "test-123"
    assert state.current_state == ConfiguratorState.POWER_SOURCE_SELECTION
    assert state.conversation_history == []
    assert state.language == "en"

    Integration Tests (tests/integration/)¶

    Purpose: Test multiple components working together

    Characteristics: - üêå Medium speed: < 5s per test - üîó Connected: Uses real services (Redis, databases) - üß© Multi-component: Tests component interactions - üìä Realistic: Closer to production scenarios

    What to test: - API endpoint functionality - Session management across services - Database operations (with test database) - Service-to-service communication - Configuration loading

    What NOT to test: - Complete end-to-end user workflows (use E2E tests) - Fine-grained business logic (use unit tests)

import pytest

@pytest.mark.integration
@pytest.mark.api
@pytest.mark.asyncio
async def test_message_endpoint_creates_new_session(api_client):
    """Test POST /message creates new session when session_id is null"""
    response = await api_client.post(
        "/api/v1/configurator/message",
        json={

            "language": "en"
        }
    )

    assert response.status_code == 200
    data = response.json()
    assert "session_id" in data
    assert data["current_state"] == "power_source_selection"

    End-to-End Tests (tests/e2e/)¶

    Purpose: Test complete user workflows from start to finish

    Characteristics: - üê¢ Slow: > 5s per test - üåê Comprehensive: Tests entire system - üë§ User-focused: Simulates real user behavior - üé≠ Scenario-based: Complete workflows

    What to test: - Complete configuration workflows (S1‚ÜíSN) - Multi-language user journeys - Error recovery scenarios - State transitions across entire flow - Real-world use cases

    What NOT to test: - Individual component logic (use unit tests) - Basic API functionality (use integration tests)

import pytest

@pytest.mark.e2e
@pytest.mark.slow
@pytest.mark.requires_neo4j
@pytest.mark.asyncio
async def test_complete_workflow_all_components(api_client):
    """Test complete workflow selecting all components"""
    # S1: Select power source
    response1 = await api_client.post(
        "/api/v1/configurator/message",
        json={"message": "I need Aristo 500ix", "language": "en"}
    )
    session_id = response1.json()["session_id"]

    # Select first product
    await api_client.post(
        "/api/v1/configurator/select",
        json={
            "session_id": session_id,
            "component": "PowerSource",
            "gin": "0446200880",
            "language": "en"
        }
    )

    # S2-S7: Continue through all states...
    # ... (complete workflow testing)

    # Verify final configuration
    final_state = await api_client.get(f"/api/v1/configurator/state/{session_id}")
    assert final_state.json()["current_state"] == "finalize"

    Best Practices¶

    1. Test Isolation¶

    ‚úÖ Good: Each test is independent 

    1
2
3
4
5
    @pytest.mark.unit
def test_function_a(mock_service):
    # Fresh mock for this test
    result = function_a(mock_service)
    assert result == expected

    ‚ùå Bad: Tests depend on execution order 

    1
2
3
4
5
6
7
8
9
    shared_state = None

def test_step_1():
    global shared_state
    shared_state = initialize()

def test_step_2():
    # Fails if test_step_1 didn't run first
    assert shared_state is not None

    2. Clear Test Names¶

    ‚úÖ Good: Descriptive names 

    1
2
3
4
5
    def test_session_storage_saves_new_session_to_redis(self):
    pass

def test_parameter_extractor_handles_mig_process_extraction(self):
    pass

    ‚ùå Bad: Vague names 

    1
2
3
4
5
    def test_session_1(self):
    pass

def test_works(self):
    pass

    3. One Assertion Concept Per Test¶

    ‚úÖ Good: Test one thing 

    1
2
3
4
5
6
7
    def test_conversation_state_initializes_with_default_language(self):
    state = ConversationState(session_id="test")
    assert state.language == "en"

def test_conversation_state_initializes_with_empty_history(self):
    state = ConversationState(session_id="test")
    assert state.conversation_history == []

    ‚ùå Bad: Multiple unrelated assertions 

    1
2
3
4
5
6
    def test_conversation_state(self):
    state = ConversationState(session_id="test")
    assert state.language == "en"
    assert state.conversation_history == []
    assert state.master_parameters == {}
    # Testing too many things

    4. Use Fixtures for Setup¶

    ‚úÖ Good: Use fixtures 

    1
2
3
4
5
6
7
8
    @pytest.fixture
def conversation_state():
    return ConversationState(session_id="test")

def test_state_transition(conversation_state):
    # Clean and readable
    next_state = conversation_state.get_next_state()
    assert next_state == ConfiguratorState.FEEDER_SELECTION

    ‚ùå Bad: Setup in every test 

    1
2
3
4
    def test_state_transition():
    state = ConversationState(session_id="test")
    state.current_state = ConfiguratorState.POWER_SOURCE_SELECTION
    # Repeated setup code

    5. Test Edge Cases¶

    Always test: - ‚úÖ Happy path (normal operation) - ‚úÖ Edge cases (empty, None, zero, max values) - ‚úÖ Error conditions (invalid input, exceptions) - ‚úÖ Boundary conditions (min/max, first/last)

    1
2
3
4
5
6
7
8
    def test_process_message_with_empty_string():
    result = process_message("")
    assert result.error == "Message cannot be empty"

def test_process_message_with_very_long_string():
    long_message = "x" * 10000
    result = process_message(long_message)
    assert result.truncated is True

    6. Don't Test Implementation Details¶

    ‚úÖ Good: Test behavior 

    1
2
3
4
    def test_user_can_select_power_source():
    response = select_product(session_id, gin)
    assert response.product_selected
    assert response.state_advanced

    ‚ùå Bad: Test implementation 

    1
2
3
    def test_select_product_calls_update_method():
    # Too tightly coupled to implementation
    assert mock_service.update.called

        # Assert
