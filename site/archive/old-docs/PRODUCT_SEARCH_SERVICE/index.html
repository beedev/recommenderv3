    Overview¶

    The Neo4j Product Search Service is Agent 2 in the 3-agent configurator architecture. It provides intelligent product search capabilities with compatibility validation, priority ranking, and LLM-based filtering.

    Purpose¶

    • Search welding equipment products stored in Neo4j graph database
    • Validate component compatibility using COMPATIBLE_WITH relationships
    • Rank results by priority scores for optimal recommendations
    • Support both sequential (S1→S7) and compound request flows

    Key Features¶

    ✅ Compatibility Validation: Uses Neo4j graph relationships to ensure component compatibility ✅ Priority Ranking: Orders results by relationship priority scores ✅ LLM Search Term Filtering: Dynamic Cypher WHERE clause generation from user intent ✅ Fuzzy Matching: Product name normalization for PowerSource, Feeder, and Cooler ✅ Fallback Logic: Shows all compatible products if no search term matches ✅ Async Architecture: Built on Neo4j async driver for high performance

) -> SearchResults

    Parameters: - response_json["PowerSource"]["gin"]: Selected PowerSource GIN (REQUIRED) - response_json["Feeder"]["gin"]: Selected Feeder GIN (OPTIONAL) - master_parameters["interconnector"]: Dict with keys like cable_length, current_capacity - limit: Maximum results (default: 3)

    Features: - Triple Compatibility: PowerSource (mandatory) + Feeder (optional) + Cooler (NOT checked) - Dynamic Priority Calculation: Sums priorities from PowerSource and Feeder relationships - Exclusion Logic: If Feeder not selected, excludes interconnectors that require Feeder

    Cypher Query Pattern:

OPTIONAL MATCH (feeder)-[r2:COMPATIBLE_WITH]->(target)  -- Optional feeder compatibility
WITH target, MIN(COALESCE(r1.priority, 999999)) AS p1, MIN(COALESCE(r2.priority, 999999)) AS p2
WITH target, (p1 + p2) AS best_priority
ORDER BY best_priority ASC
LIMIT $limit

    Search Features¶

    1. Compatibility Validation¶

    Mechanism: Uses Neo4j COMPATIBLE_WITH bidirectional relationships.

    Example Graph Structure: 

    1
2
    (Aristo 500ix:PowerSource)-[:COMPATIBLE_WITH {priority: 1}]-(RobustFeed U6:Feeder)
(Aristo 500ix:PowerSource)-[:COMPATIBLE_WITH {priority: 2}]-(RobustFeed U4:Feeder)

    Validation Logic: - Downstream components (Feeder, Cooler, etc.) MUST have COMPATIBLE_WITH relationship to selected PowerSource - Interconnectors MUST be compatible with PowerSource + Feeder (if selected) - Torches MUST be compatible with PowerSource + Feeder + Cooler (if selected)

    2. Priority Ranking¶

    Purpose: Order results by quality/relevance scores.

    Priority Sources: - Relationship property: r.priority (lower = better) - Default priority: 999999 (if not set)

    Aggregation Logic: 

    1
2
3
4
5
    # Single relationship (Feeder)
best_priority = MIN(COALESCE(r.priority, 999999))

# Multiple relationships (Torch with PS + Feeder + Cooler)
best_priority = p1 + p2 + p3

    Example Priorities: - Priority 1: Best match (e.g., manufacturer recommended) - Priority 2: Good match (e.g., tested compatibility) - Priority 3+: Acceptable match - Priority 999999: No priority set (sorted by name)

    3. LLM Search Term Filtering¶

    Purpose: Apply user-specified requirements as dynamic WHERE clauses.

    Process: 1. Extract Search Terms from master_parameters using _build_search_terms_from_component() - Material (e.g., "stainless steel", "aluminum") - Application (e.g., "automotive", "shipbuilding") - Process (e.g., "MIG", "TIG") - Product name (e.g., "Aristo 500") - Technical specs (e.g., "500A", "10m cable")

    1. Generate WHERE Clause using _add_search_term_filters() 

      1
2
3
4
      WHERE (toLower(f.description_catalogue) CONTAINS toLower($term_0)
    OR toLower(f.description_ruleset) CONTAINS toLower($term_0)
    OR toLower(f.item_name) CONTAINS toLower($term_0))
  AND (toLower(f.description_catalogue) CONTAINS toLower($term_1) ...)

    2. Execute with Fallback: If no results, remove search terms and show all compatible products

    Example: 

    1
2
3
4
5
6
7
8
9
    master_parameters = {
    "feeder": {
        "material": "aluminum",
        "cooling_type": "water-cooled"
    }
}

# Search terms: [" aluminum", " water-cooled"]
# Cypher WHERE: ... CONTAINS ' aluminum' OR ... CONTAINS ' water-cooled'

    4. Fuzzy Matching¶

    Purpose: Handle user typos and variations in product names.

    Technology: RapidFuzz library with token-based matching

    Supported Components: PowerSource, Feeder, Cooler

    Process: 1. Extract first word from user input (e.g., "Aristo" from "Aristo 500ix") 2. Normalize: Remove non-alphanumeric, lowercase 3. Check for exact base match: If multiple products share same base name, return base (e.g., "Aristo") 4. Fuzzy match: Score cutoff 80% 5. Return matched name or original input

    Example:

# Known products: ["Aristo 500ix", "Aristo 400ix"]
# Base match: "Aristo" (returns first word for disambiguation)

# User input: "Aristoo 500"  (typo)
# Fuzzy match: "Aristo 500ix" (score: 92%)

    Code Location: _normalize_product_name() method (lines 80-116)

    5. Fallback Logic¶

    Purpose: Always show results, even if search terms don't match.

    Mechanism: - Primary Query: Search with LLM-generated search terms - Fallback Query: Same query without search terms (all compatible) - Execution: _execute_search_with_fallback() tries primary first, then fallback

    Purpose: Match both decimal and non-decimal measurement formats.

    Process: Convert "10m" to ["10m", "10.0m"] for better matching.

    Performance Considerations¶

    1. Query Optimization¶

    Index Recommendations: 

    1
2
3
    CREATE INDEX product_gin FOR (p:Product) ON (p.gin);
CREATE INDEX product_category FOR (p:Product) ON (p.category);
CREATE INDEX product_name FOR (p:Product) ON (p.item_name);

    Best Practices: - Use LIMIT on all queries - Filter by category early in query - Use WITH clauses to reduce intermediate results - Prefer COALESCE(r.priority, 999999) over IS NULL checks

    2. Async Architecture¶

    Benefits: - Non-blocking I/O for Neo4j queries - Concurrent search for compound requests - Scalable to 100+ concurrent users

    Usage: 

    1
2
3
4
5
6
7
8
9
    # Sequential (slow)
ps_results = await neo4j_search.search_power_source(...)
feeder_results = await neo4j_search.search_feeder(...)

# Concurrent (fast) - for compound requests
import asyncio
ps_task = neo4j_search.search_power_source(...)
feeder_task = neo4j_search.search_feeder(...)
ps_results, feeder_results = await asyncio.gather(ps_task, feeder_task)

    3. Caching¶

    Product Names Cache: - Loaded once on initialization from product_names.json - Used for fuzzy matching (PowerSource, Feeder, Cooler only) - Reduces file I/O during searches

    Future Enhancement: Add Redis caching for frequent searches

    4. Connection Pooling¶

    Singleton Pattern: 

    1
2
3
4
5
6
7
    _neo4j_search = None

async def get_neo4j_search(uri: str, username: str, password: str):
    global _neo4j_search
    if _neo4j_search is None:
        _neo4j_search = Neo4jProductSearch(uri, username, password)
    return _neo4j_search

    Benefits: - Reuses single Neo4j driver instance - Connection pooling handled by driver - Reduces connection overhead
