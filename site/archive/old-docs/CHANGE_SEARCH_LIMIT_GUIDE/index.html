    Guide: Change Search Result Limit from 3 to 10¶

    File: src/backend/app/services/orchestrator/state_orchestrator.py

    Current Limit: 3 products per search Target Limit: 10 products per search

    Overview¶

    The orchestrator currently limits search results to 3 products in two contexts: 1. Regular State Handler Searches - When user is in a specific state (e.g., selecting feeder) 2. Proactive Searches - After a selection, searching next component automatically

    All limits are currently hardcoded as either: - limit=3 (13 occurrences) - PROACTIVE_LIMIT = 3 (3 local definitions in functions)

    Recommended Approach: Create Class-Level Constant¶

    Benefits: - ‚úÖ Single source of truth (change once, applies everywhere) - ‚úÖ Easy to configure and maintain - ‚úÖ Self-documenting code - ‚úÖ Easy to make different limits for regular vs proactive searches if needed

    Alternative: Find and replace all limit=3 with limit=10 (not recommended - harder to maintain)

    Performance Considerations¶

    Database Load¶

    • Before: Fetching 3 products per search
    • After: Fetching 10 products per search
    • Impact: ~3.3x more data per query

    Mitigation: - Neo4j queries are fast (typically < 100ms) - Increase is minimal for graph database - Monitor query times in production

    Response Size¶

    • Before: ~1KB per product √ó 3 = ~3KB
    • After: ~1KB per product √ó 10 = ~10KB
    • Impact: ~3.3x larger JSON responses

    Mitigation: - Still small for modern networks (10KB is trivial) - Consider compression if needed (gzip can reduce by 70%)

    User Experience¶

    • Positive: More choices, less need to refine search
    • Negative: More scrolling, could be overwhelming

    Recommendation: - Start with 10 and monitor user feedback - Can implement pagination if needed ("Show more" button) - Consider making limit configurable via API parameter in future

    Questions & Answers¶

    Q: Why not just use 10 everywhere with find-replace?¶

    A: Class constants are better because: - Single source of truth (change once) - Self-documenting code - Easy to have different limits for different contexts - Easy to make configurable later

    Q: Should regular and proactive searches have different limits?¶

    A: Current recommendation: Both at 10. But you could: - Regular: 10 (user actively searching) - Proactive: 5 (just showing preview of next step) This is why we have two constants.

    Q: Will this slow down the application?¶

    A: Minimal impact: - Neo4j graph queries are very fast (< 100ms) - 10 products is still small data (~10KB) - LLM processing time dominates (1-3 seconds)

    Q: Can I make it higher than 10?¶

    A: Yes, but consider: - 10 is a good UX balance (not overwhelming) - Beyond 15-20, users may not scroll - Consider pagination ("Show more") for larger sets

    Q: What if a search returns < 10 products?¶

    A: No problem: - Limit is a MAX, not a requirement - If only 5 match, shows all 5 - If 0 match, shows compatibility fallback message
