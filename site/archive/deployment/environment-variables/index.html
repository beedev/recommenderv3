    Environment Configuration¶

    This document describes environment variable configuration for different deployment scenarios.

    Quick Start¶

    For Local Development (Docker Compose)¶

cp deployment/env/.env.development.example src/backend/.env

# Edit the file and update:
# - OPENAI_API_KEY (required)
# - LANGSMITH_API_KEY (optional, for debugging)

# Start all services
cd deployment/docker

    Available Templates¶

    .env.example¶

    Complete template with all available environment variables.

    Use for: - Reference documentation - Understanding all configuration options - Custom deployment scenarios

    Key sections: - Application configuration - OpenAI LLM settings - Database connections (Neo4j, PostgreSQL, Redis) - LangSmith observability - Logging and CORS - Feature flags - Production optimizations

    .env.development.example¶

    Optimized for local development with Docker Compose.

    Features: - Debug mode enabled - Single worker (for easier debugging) - Auto-reload on code changes - Docker service names for databases (neo4j, postgres, redis) - Weak passwords (OK for local development) - Detailed error pages - SQL query logging

    Usage: 

    1
2
3
4
    cp deployment/env/.env.development.example src/backend/.env
# Edit OPENAI_API_KEY
cd deployment/docker
docker-compose up -d

    .env.production.example¶

    Optimized for production deployment with systemd.

    Features: - Debug mode disabled - Multiple workers (9 for 4-core machine) - No auto-reload - Production database connection strings - Strong password placeholders - Rate limiting - LangSmith monitoring enabled - Backup configuration

    Usage: 

    1
2
3
4
    cp deployment/env/.env.production.example src/backend/.env
# Update ALL passwords and secrets
chmod 600 src/backend/.env
sudo ./deployment/deploy.sh install

    Environment Variables Reference¶

    Required Variables¶

    These must be set for the application to run:

    Variable Description Example
    OPENAI_API_KEY OpenAI API key for LLM agents sk-proj-xxx...
    NEO4J_URI Neo4j connection URI bolt://localhost:7687
    NEO4J_USERNAME Neo4j username neo4j
    NEO4J_PASSWORD Neo4j password your_password
    POSTGRES_HOST PostgreSQL host localhost
    POSTGRES_DB PostgreSQL database name pconfig
    POSTGRES_USER PostgreSQL username postgres
    POSTGRES_PASSWORD PostgreSQL password your_password
    REDIS_HOST Redis host localhost
    REDIS_PASSWORD Redis password your_password

    Optional Variables¶

    Variable Description Default
    ENVIRONMENT Environment mode development
    DEBUG Enable debug mode false
    PORT Application port 8000
    WORKERS Number of uvicorn workers 4
    CACHE_TTL Session TTL in seconds 3600
    LOG_LEVEL Logging level INFO
    CORS_ORIGINS Allowed CORS origins http://localhost:3000
    LANGSMITH_TRACING Enable LangSmith false

    See deployment/env/.env.example for the complete list with descriptions.

    Database hosts use actual hostnames/IPs:

    Security Best Practices¶

    Passwords¶

    Development: - Simple passwords are OK for local development - Use defaults from templates

    Production: - Generate strong random passwords (20+ characters) - Use different passwords for each service - Rotate passwords regularly

    Generate strong passwords: 

    1
2
3
4
5
6
7
8
    # Python method
python -c "import secrets; print(secrets.token_urlsafe(32))"

# OpenSSL method
openssl rand -base64 32

# Linux method
tr -dc A-Za-z0-9 </dev/urandom | head -c 32; echo

    Secret Management¶

    Bad (never do this): 

    1
2
    # Committing .env to git
git add .env  # ❌ NEVER DO THIS

    Good (recommended approaches):

    1. Environment variables in systemd: 

      1
2
      [Service]
EnvironmentFile=/path/to/.env

    2. Secret management tools:

    3. HashiCorp Vault
    4. AWS Secrets Manager
    5. Azure Key Vault

    6. Google Secret Manager

    7. File permissions: 

      1
2
      chmod 600 .env
chown appuser:appgroup .env

    What to Change in Production¶

    Always change: - SECRET_KEY - Application secret - JWT_SECRET_KEY - JWT signing key - OPENAI_API_KEY - Use production/organization key - NEO4J_PASSWORD - Neo4j database password - POSTGRES_PASSWORD - PostgreSQL password - REDIS_PASSWORD - Redis password - CORS_ORIGINS - Production domain whitelist

    Review and adjust: - WORKERS - Based on CPU cores - CACHE_TTL - Session timeout (default 1 hour) - RATE_LIMIT - API rate limiting - RETENTION_DAYS - Backup retention policy - LOG_LEVEL - Set to INFO or WARNING in production

    Optional (add if using): - LANGSMITH_API_KEY - For production monitoring - SENTRY_DSN - For error tracking - NEW_RELIC_LICENSE_KEY - For APM

    Troubleshooting¶

    Application won't start¶

    Check environment file: 

    1
2
3
4
5
6
    # Verify file exists
ls -la src/backend/.env

# Check permissions
# Should be readable by app user
chmod 644 src/backend/.env  # or 600 for stricter security

    Validate required variables: 

    1
2
3
4
5
    # Check if required vars are set
grep OPENAI_API_KEY src/backend/.env
grep NEO4J_URI src/backend/.env
grep POSTGRES_HOST src/backend/.env
grep REDIS_HOST src/backend/.env

    Test environment loading: 

    1
2
    cd src/backend
python -c "from dotenv import load_dotenv; import os; load_dotenv(); print(os.getenv('OPENAI_API_KEY'))"

    Database connection errors¶

    Development (Docker Compose): 

    1
2
3
    # Use service names, not localhost
NEO4J_URI=bolt://neo4j:7687  # ✅ Correct
NEO4J_URI=bolt://localhost:7687  # ❌ Wrong inside Docker

    Production: 

    1
2
3
    # Use actual hostnames/IPs
NEO4J_URI=bolt://your-server.com:7687  # ✅ Correct
POSTGRES_HOST=your-postgres.database.azure.com  # ✅ Correct

    Test connections: 

    1
2
3
4
5
6
7
8
    # Neo4j
cypher-shell -a $NEO4J_URI -u $NEO4J_USERNAME -p $NEO4J_PASSWORD "RETURN 1;"

# PostgreSQL
PGPASSWORD=$POSTGRES_PASSWORD psql -h $POSTGRES_HOST -U $POSTGRES_USER -d $POSTGRES_DB -c "SELECT 1;"

# Redis
redis-cli -h $REDIS_HOST -p $REDIS_PORT -a $REDIS_PASSWORD PING

    OpenAI API errors¶

    Check API key: 

    1
2
    # Key should start with sk-proj- or sk-
grep OPENAI_API_KEY src/backend/.env

    Test API key: 

    1
2
    curl https://api.openai.com/v1/models \
  -H "Authorization: Bearer $OPENAI_API_KEY"

    Common issues: - Incorrect key format - Key revoked or expired - Rate limit exceeded - Insufficient credits

    CORS errors in browser¶

    Development: 

    1
2
    # Allow localhost on all ports
CORS_ORIGINS=http://localhost:3000,http://localhost:3001,http://127.0.0.1:3000

    Production: 

    1
2
    # Whitelist specific domains only
CORS_ORIGINS=https://yourdomain.com,https://www.yourdomain.com

    Environment Variable Precedence¶

    When the same variable is defined in multiple places:

    1. Environment variables (highest priority)
    2. Set in shell: export OPENAI_API_KEY=xxx

    3. Set in systemd: Environment=OPENAI_API_KEY=xxx

    4. .env file (medium priority)

    5. src/backend/.env

    6. Default values in code (lowest priority)

    7. Defined in app/main.py or config files

    Example: 

    1
2
3
4
5
6
7
    # In .env
OPENAI_API_KEY=sk-test-key

# In shell
export OPENAI_API_KEY=sk-prod-key

# Application will use: sk-prod-key (shell env wins)

    Migration from Old Setup¶

    If migrating from an existing deployment:

    1. Locate current .env file: 

      1
      find . -name ".env" -type f

    2. Compare with templates: 

      1
      diff src/backend/.env deployment/env/.env.production.example

    3. Add missing variables: 

      1
2
3
4
      # Check for new required variables in template
grep -v "^#" deployment/env/.env.production.example | grep "=" | cut -d= -f1 | sort > /tmp/template_vars
grep -v "^#" src/backend/.env | grep "=" | cut -d= -f1 | sort > /tmp/current_vars
comm -13 /tmp/current_vars /tmp/template_vars  # Shows missing vars

    4. Update systemd service (if using): 

      1
2
      # Ensure service file references .env
grep EnvironmentFile /etc/systemd/system/esab-recommender.service
