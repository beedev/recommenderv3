    Redis Configuration¶

    What is Redis?¶

    Redis provides: - ‚úÖ Session Persistence - Sessions survive backend restarts - ‚úÖ Distributed Caching - Share cache across multiple instances - ‚úÖ Performance - Fast in-memory data store

CACHE_TTL=3600

    When to Enable Redis¶

    ‚úÖ Enable if you need: - üî¥ Production deployments - üî¥ Multiple backend instances (horizontal scaling) - üî¥ Sessions must survive backend restarts - üî¥ Distributed caching across instances

    ‚ö†Ô∏è Disable if: - üü¢ Development/testing - üü¢ Single backend instance - üü¢ In-memory sessions are acceptable - üü¢ No scaling requirements

    What is LangSmith?¶

    LangSmith provides: - ‚úÖ Workflow Tracing - Track complete request flows through all agents - ‚úÖ Performance Metrics - Monitor LLM call duration, tokens, costs - ‚úÖ Error Tracking - Capture and analyze failed operations - ‚úÖ Dashboard - Visualize workflows at https://smith.langchain.com

    Comparison Table¶

    Configuration Comparison¶

    Aspect Redis LangSmith
    Purpose Session persistence & caching Observability & monitoring
    Environment Variable ENABLE_REDIS_CACHING LANGSMITH_TRACING
    Values true/false true/false
    Default false (in-memory fallback) false (disabled)
    Requires API Key No Yes (but optional)
    Requires Cloud Service Optional (can be local) Yes (LangSmith cloud)
    Impact if Down Sessions in memory (acceptable) No monitoring (acceptable)
    Affects Performance Yes (positive) No (monitoring only)
    Restart Type Full rebuild Simple restart
    Data Loss Risk Yes, if not persisted No (only monitoring)
    Health Check /health shows status /health shows status

    Restart Requirements¶

    Service Configuration Change Required Action
    Redis ENABLE_REDIS_CACHING changed docker compose down -v && up --build -d
    LangSmith LANGSMITH_TRACING changed docker compose restart backend

    Why different? - Redis involves data structure changes (requires rebuild) - LangSmith is just a flag (simple restart works)

    Performance Impact¶

    With Redis Enabled¶

    • ‚úÖ Faster session lookups (Redis is in-memory, very fast)
    • ‚úÖ Distributed session sharing across instances
    • ‚úÖ Persistent sessions across restarts
    • ‚ö†Ô∏è Extra network call to Redis for each request

    Without Redis¶

    • ‚úÖ One less dependency
    • ‚úÖ No network overhead
    • ‚ö†Ô∏è Slower (application memory, not optimized)
    • ‚ö†Ô∏è Sessions lost on restart
    • ‚ö†Ô∏è Can't scale to multiple instances

    With LangSmith Enabled¶

    • ‚úÖ Complete workflow visibility
    • ‚úÖ Performance metrics
    • ‚ö†Ô∏è Extra network call to LangSmith (async, non-blocking)
    • ‚ö†Ô∏è Slightly increased latency for logging

    Without LangSmith¶

    • ‚úÖ No external monitoring calls
    • ‚úÖ Slightly lower latency
    • ‚ö†Ô∏è No visibility into agent actions
    • ‚ö†Ô∏è Harder to debug issues
