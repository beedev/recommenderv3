    Docker Compose Production - Change Comparison¶

    Overview¶

    This document highlights the specific changes made to align docker-compose.prod.yml with the updated docker-compose.yml structure while maintaining production-appropriate configurations.

    üÜï New Services Added¶

    # OLD: Redis was assumed to be external (REDIS_HOST variable)
# NEW: Redis runs locally in Docker with production configuration

redis:
  image: redis:7-alpine
  container_name: esab-redis-prod
  ports:
    - "${REDIS_PORT:-6379}:6379"
  volumes:
    - redis-data:/data
  command: >
    redis-server
    --requirepass ${REDIS_PASSWORD:-esab_redis_prod_password}
    --maxmemory 1gb                    # ‚¨ÜÔ∏è Increased from 512mb (dev)
    --maxmemory-policy allkeys-lru
    --appendonly yes
    --appendfsync everysec
    --save 900 1                        # üÜï RDB snapshots for persistence
    --save 300 10
    --save 60 10000

    Why this change? - Better performance with local caching - Simplified deployment (one less external dependency) - Full control over Redis configuration - Cost-effective for caching workloads

    üöÄ Migration Path¶

    From Old to New Deployment¶

    1. Backup existing Redis data (if using external Redis): 

      1
2
      # Backup from your current Redis instance
redis-cli --rdb /backup/dump.rdb

    2. Update environment variables: 

      1
2
3
4
5
6
      # Remove or comment out (no longer needed):
# REDIS_HOST=external-redis.example.com
# REDIS_PORT=6379

# Keep or add:
REDIS_PASSWORD=your_new_secure_password

    3. Deploy new stack: 

      1
2
      docker-compose -f docker-compose.prod.yml down
docker-compose -f docker-compose.prod.yml up -d

    4. Verify services: 

      1
2
      docker-compose -f docker-compose.prod.yml ps
curl http://localhost:8000/health

    5. Optional: Import Redis data: 

      1
2
      docker cp dump.rdb esab-redis-prod:/data/
docker-compose -f docker-compose.prod.yml restart redis

    ‚ö†Ô∏è Breaking Changes¶

    Configuration Changes Required¶

    1. Redis Connection
    2. If you were using external Redis, you'll need to migrate data

    3. Update any external monitoring tools pointing to Redis

    4. New Ports Used

    5. Port 6379: Redis (if exposing externally)

    6. Port 8001: RedisInsight UI

    7. Volume Management

    8. New volumes created for persistence
    9. Plan backup strategy for Redis data

    Non-Breaking Changes¶

    • All Neo4j and PostgreSQL configurations remain identical
    • Backend API endpoints unchanged
    • Environment variable structure mostly unchanged
    • Can still use external Redis by modifying REDIS_HOST

    üéì Best Practices Applied¶

    1. ‚úÖ Service Health Dependencies - Backend waits for Redis to be healthy
    2. ‚úÖ Named Volumes - Clear, production-specific naming
    3. ‚úÖ Resource Limits - All services have defined limits
    4. ‚úÖ Logging - Consistent log rotation across all services
    5. ‚úÖ Persistence - Redis configured with both AOF and RDB
    6. ‚úÖ Monitoring - Built-in UI for cache inspection
    7. ‚úÖ Security - Password-protected Redis with fallback defaults
    8. ‚úÖ Documentation - Comprehensive inline comments
