    Lucene Score Filtering & Score Metadata Implementation Plan¶

    Date: 2025-11-07 Status: Planning Phase Goal: Add intelligent score-based filtering (top 25%) and score metadata to Lucene search results without disrupting frontend

    üìã Executive Summary¶

    Enhance the Lucene search implementation with: 1. ‚úÖ Score-based filtering: Return only products within 25% of top score 2. ‚úÖ Score metadata: Add optional score field to ProductResult without breaking frontend 3. ‚úÖ Backward compatibility: Traditional search (no scores) continues to work 4. ‚úÖ Frontend-ready: Score available for display but optional

    üéØ Requirements¶

    Requirement 1: Top 25% Score Filtering¶

    Business Logic: - If top product has score = 1.0, return products with score >= 0.75 - If top product has score = 0.8, return products with score >= 0.60 - Formula: min_score_threshold = top_score * 0.75 - Applies AFTER min_score filter (0.5 default)

    async def search_power_source_lucene(
    self,
    user_message: str,
    limit: int = 10,
    offset: int = 0,
    min_score: float = 0.5,
    score_threshold_percent: float = 0.75  # üÜï NEW: Top 25% filter
) -> SearchResults:
    """
    Full-text search with intelligent score-based filtering.

    Args:
        score_threshold_percent: Return products within this % of top score (0.75 = top 25%)
    """
    # Execute Lucene query
    records = await session.run(query, params)

    # Extract products with scores
    products_with_scores = []
    for record in records:
        product = ProductResult(
            gin=record["gin"],
            name=record["name"],
            category=record["category"],
            description=record.get("description"),
            specifications=specs,
            score=record["score"]  # üÜï Store score
        )
        products_with_scores.append((product, record["score"]))

    # üÜï STEP 1: Find top score
    if not products_with_scores:
        return SearchResults(products=[], total_count=0, ...)

    top_score = products_with_scores[0][1]  # Already sorted DESC by score

    # üÜï STEP 2: Filter by score threshold (top 25%)
    min_threshold = top_score * score_threshold_percent
    filtered_products = [
        product for product, score in products_with_scores
        if score >= min_threshold
    ]

    logger.info(f"Score filtering: top={top_score:.3f}, threshold={min_threshold:.3f}, kept {len(filtered_products)}/{len(products_with_scores)}")

    # Apply pagination AFTER filtering
    has_more = len(filtered_products) > limit
    if has_more:
        filtered_products = filtered_products[:limit]

    return SearchResults(
        products=filtered_products,
        total_count=len(filtered_products),
        filters_applied={
            "search_method": "lucene_fulltext",
            "top_score": top_score,
            "score_threshold": min_threshold,
            "products_filtered": len(products_with_scores) - len(filtered_products)
        },
        ...
    )

    üö® Risk Mitigation¶

    Risk 1: Too Aggressive Filtering (No Results)¶

    Scenario: Score threshold filters out ALL products

    Mitigation: 

    1
2
3
4
5
    # Add minimum result guarantee
if len(filtered_products) < 3 and len(products_with_scores) >= 3:
    logger.warning(f"Score filter too aggressive, relaxing threshold")
    # Return top 3 regardless of score
    filtered_products = [p for p, _ in products_with_scores[:3]]

    Detection: Monitor logs for "too aggressive" warnings

    Recovery: Lower score_threshold_percent in config (e.g., 0.5 for top 50%)

    Risk 2: Score Field Breaks Frontend¶

    Mitigation: score is Optional[float] = None (backward compatible)

    Detection: Frontend automated tests, browser console errors

    Recovery: Frontend ignores unknown fields by default (no action needed)

    Risk 3: Performance Impact¶

    Mitigation: Filtering is O(n) in-memory operation (fast)

    Detection: Monitor response times, P95 latency

    Recovery: Disable score filtering with score_threshold_percent: 0.0
