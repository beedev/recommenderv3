    Dynamic State Machine Architecture¶

    Complete Technical Overview - S1→SN Configuration-Driven Flow

    Overview¶

    The ESAB Welding Equipment Configurator uses a fully dynamic, configuration-driven state machine that allows adding new states (S8, S9, ..., SN) without modifying Python code.

    Key Features¶

    ✅ Zero-Code State Addition - Add states via JSON configuration only ✅ Dynamic Enum Generation - ConfiguratorState enum created at runtime ✅ Processor Registry Pattern - Automatic processor assignment ✅ Config-Driven Prompts - All user messages from JSON ✅ Runtime Validation - Health checks and schema validation ✅ Backwards Compatible - Existing code continues to work

    StateProcessorRegistry
├── power_source_selection → PowerSourceProcessor
├── feeder_selection → SingleSelectionProcessor
├── cooler_selection → SingleSelectionProcessor
├── interconnector_selection → SingleSelectionProcessor
├── torch_selection → SingleSelectionProcessor
└── accessories_selection → MultiSelectionProcessor

    Processor Types:

    1. SingleSelectionProcessor
    2. For: Feeder, Cooler, Interconnector, Torch

    3. Handles: Single product selection, explicit index, skip

    4. MultiSelectionProcessor

    5. For: Accessories

    6. Handles: Multiple selections, "done" command, list building

    7. PowerSourceProcessor

    8. For: Power source (S1)

    9. Handles: Applicability loading, mandatory selection

    10. Custom Processors

    11. For: Complex logic
    12. Requires: Custom class implementing BaseStateProcessor

    logger.info("✓ All services initialized")

    yield

    # Shutdown
    await close_redis()
    await close_postgresql()

    Performance Considerations¶

    Caching Strategy¶

    StateFactory: - Enum cached after first creation - Metadata cached per state - Clear cache: StateFactory.clear_cache()

    ConfigMonitor: - Health results cached - Clear cache: monitor.clear_cache() - API endpoint: POST /api/v1/health/config/cache/clear

    ConfigurationService: - LRU cache for config loading - TTL-based invalidation - Automatic reload on file change (optional)

    Optimization Tips¶

    1. Minimize config reloads: Cache aggressively
    2. Use health check cache: Don't validate on every request
    3. Lazy processor loading: Only load when needed
    4. State sequence caching: Load once at startup
