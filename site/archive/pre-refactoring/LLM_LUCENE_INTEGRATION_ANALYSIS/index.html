    LLM-Lucene Integration & Constraint-Based Filtering Analysis¶

    Date: 2025-01-09 Session Focus: Operator-based constraint filtering for Neo4j Lucene search Status: Investigation Complete, Implementation Rolled Back Version: 1.0

    Executive Summary¶

    This document details the work done to integrate LLM-based parameter extraction with Neo4j Lucene full-text search and constraint-based filtering using operator logic (≤, ≥, <, >, =, range, approx).

    What We Built: - ✅ LLM prompt enhancements for operator extraction - ✅ Dual-mode numeric constraint matching (dict + string formats) - ✅ Feature-based post-filtering pipeline - ✅ Test infrastructure for validation

    What We Discovered: - ❌ Schema is documentation-only (not enforced at runtime) - ❌ Feature key mismatch (current_rating vs current_output) - ❌ Score threshold filtering too aggressive (separate issue) - ❌ No validation pipeline for extracted parameters

    Outcome: - Investigation complete with comprehensive documentation - Implementation rolled back (not production-ready) - Clear path forward documented for future implementation

    def _build_search_terms(self, component_dict: Dict[str, Any]) -> List[str]:
    """
    Builds Lucene search terms from extracted parameters

    Technical fields that map to Lucene search:
    - thickness → "3mm", "thick 3mm"
    - current_output → "500A", "500 A", "500 Amps"
    - voltage → "380V", "dual voltage"
    - duty_cycle → "60%", "100%"
    - cable_length → "3m", "5 meters"
    - wire_diameter → "1.0mm", "1.2mm"
    """
    search_terms = []

    # Extract technical specifications
    technical_fields = ["thickness", "current_output", "voltage", "duty_cycle",
                       "cable_length", "wire_diameter", "flow_rate", "cooling_type"]

    for key in technical_fields:
        value = component_dict.get(key)
        if value and isinstance(value, str):
            # Expand: "500A" → ["500A", "500 A", "500 Amps"]
            search_terms.extend(self._expand_measurement_terms(value))

    # Extract categorical features
    categorical_fields = ["process", "material", "cooling_type", "application"]
    for key in categorical_fields:
        value = component_dict.get(key)
        if value and isinstance(value, str):
            search_terms.append(value)

    return search_terms

    async def search_power_source_lucene(
    self,
    user_message: str,
    master_parameters: Dict[str, Any],
    limit: int = 10
) -> SearchResults:
    """
    Hybrid search: Lucene full-text + LLM-extracted parameters

    Pipeline:
    1. Build Lucene query from user message (stopwords removed)
    2. Execute Neo4j Lucene search (productIndex)
    3. Apply score threshold filtering (25% default)
    4. Apply feature-based post-filtering (operator constraints)
    5. Rank by priority and score
    """

    # Step 1: Normalize query for Lucene
    english_query = self._normalize_search_query(user_message)
    # "I am satisfied with a maximum output of 300 Amps"
    # → "satisfied maximum output 300 Amps"

    # Step 2: Build Lucene query
    power_source_params = master_parameters.get("power_source", {})
    search_terms = self._build_search_terms(power_source_params)

    # Combine user query + extracted search terms
    lucene_query = f"{english_query} {' '.join(search_terms)}"

    # Step 3: Execute Lucene search
    query = """
    CALL db.index.fulltext.queryNodes("productIndex", $lucene_query)
    YIELD node, score
    WHERE node.category = 'Powersource'
    RETURN node, score
    ORDER BY score DESC
    LIMIT $limit
    """

    results = await self.driver.execute_query(query, {
        "lucene_query": lucene_query,
        "limit": limit
    })

    # Step 4: Score threshold filtering (ISSUE: Too aggressive)
    products_with_scores = self._apply_score_threshold(results)

    # Step 5: Feature-based post-filtering (NEW - ISSUE: Key mismatch)
    filtered_products = self._filter_by_feature_requirements(
        products_with_scores,
        power_source_params
    )

    return SearchResults(products=filtered_products)

    Key Integration Points:

    1. LLM → Lucene Query:
    2. LLM extracts: {"current_output": "500A", "process": "MIG"}
    3. Builds query: "500A MIG GMAW welder aluminum"

    4. Lucene searches: Full-text across name + description

    5. Lucene → LLM Post-Filter:

    6. Lucene returns: All matching products with scores
    7. LLM parameters filter: Apply operator constraints
    8. Final results: Subset meeting all requirements

    def _matches_numeric_constraint(
    self,
    user_constraint: Union[str, Dict[str, Any]],
    product_value: float,
    tolerance: float = 0.2
) -> bool:
    """
    Check if product value matches user numeric constraint with operator support

    Dual-mode support:
    1. Dict format (NEW): {"value": 300, "operator": "lte", "unit": "A"}
    2. String format (LEGACY): "300A" (uses tolerance matching)

    Operators:
    - lte (≤): product_value <= user_value
    - gte (≥): product_value >= user_value
    - lt (<): product_value < user_value
    - gt (>): product_value > user_value
    - eq (=): abs(product_value - user_value) <= tolerance * user_value
    - range: min_value <= product_value <= max_value
    - approx (≈): abs(product_value - user_value) <= tolerance * user_value

    Examples:
    - user_constraint = {"value": 300, "operator": "lte"}
      product_value = 250 → True (250 ≤ 300)
      product_value = 350 → False (350 > 300)

    - user_constraint = {"min": 300, "max": 500, "operator": "range"}
      product_value = 400 → True (300 ≤ 400 ≤ 500)
      product_value = 600 → False (600 > 500)

    - user_constraint = "500A" (legacy)
      product_value = 450 → True (within 20% tolerance)
      product_value = 350 → False (outside tolerance)
    """

    # DUAL-MODE HANDLING
    if isinstance(user_constraint, dict):
        # NEW: Operator-based matching
        operator = user_constraint.get("operator", "eq")

        if operator == "lte":
            # Less Than or Equal (≤)
            user_value = user_constraint.get("value")
            return product_value <= user_value

        elif operator == "gte":
            # Greater Than or Equal (≥)
            user_value = user_constraint.get("value")
            return product_value >= user_value

        elif operator == "lt":
            # Less Than (<)
            user_value = user_constraint.get("value")
            return product_value < user_value

        elif operator == "gt":
            # Greater Than (>)
            user_value = user_constraint.get("value")
            return product_value > user_value

        elif operator == "eq":
            # Equal (=) with tolerance
            user_value = user_constraint.get("value")
            return abs(product_value - user_value) <= tolerance * user_value

        elif operator == "range":
            # Between min and max
            min_value = user_constraint.get("min")
            max_value = user_constraint.get("max")
            return min_value <= product_value <= max_value

        elif operator == "approx":
            # Approximately equal (±tolerance)
            user_value = user_constraint.get("value")
            return abs(product_value - user_value) <= tolerance * user_value

        else:
            # Unknown operator, default to approximate
            user_value = user_constraint.get("value")
            return abs(product_value - user_value) <= tolerance * user_value

    else:
        # LEGACY: String format "500A" → tolerance-based matching
        user_value = self._parse_numeric_value(user_constraint)
        if user_value is None:
            return True  # Can't parse, don't filter

        return abs(product_value - user_value) <= tolerance * user_value

    def _matches_feature_requirements(
    self,
    product: ProductResult,
    master_parameters: Dict[str, Any],
    feature_data: Dict[str, Any]
) -> bool:
    """
    Check if product specifications match user requirements based on LLM features

    Process:
    1. Extract product specifications from description
    2. Load feature definitions from category_features_llm.json
    3. For each numeric spec, check if product value matches constraint
    4. For each categorical spec, check if product value matches requirement

    Returns:
    - True if ALL requirements matched (or no requirements specified)
    - False if ANY requirement fails
    """

    features = feature_data.get("features", {})
    product_specs = self._extract_product_specs(product.description or "")

    # 1. Check categorical specifications (exact match)
    for cat_spec in features.get("categorical_specs", []):
        spec_name = cat_spec["name"].lower().replace(" ", "_")

        if spec_name in master_parameters:
            user_value = master_parameters[spec_name]
            product_value = product_specs.get(spec_name)

            if product_value and user_value.lower() not in product_value.lower():
                return False  # Categorical mismatch

    # 2. Check numeric specifications WITH OPERATOR SUPPORT (NEW)
    for num_spec in features.get("numeric_specs", []):
        spec_name = num_spec["name"].lower().replace(" ", "_")
        # "Current Output" → "current_output"

        # ISSUE: Feature key mismatch occurs here
        if spec_name not in master_parameters:
            continue  # ← SKIPS if key doesn't match!

        user_constraint = master_parameters[spec_name]
        product_value = self._parse_numeric_value(product_specs.get(spec_name))

        if product_value is not None:
            # Apply operator constraint (NEW)
            if not self._matches_numeric_constraint(user_constraint, product_value, tolerance=0.2):
                return False  # Constraint violation

    return True  # All checks passed

# Score filtering
if products_with_scores and score_threshold_percent > 0:
    top_score = max(p["score"] for p in products_with_scores)
    threshold = top_score * (1 - score_threshold_percent / 100)

    filtered_products = [
        p for p in products_with_scores
        if p["score"] >= threshold  # ← Renegade fails this check
    ]

    products_with_scores = filtered_products  # ← Renegade removed

    Lessons Learned¶

    1. Schema ≠ Runtime Behavior¶

    What We Thought: - Schema controls what LLM extracts - Changing schema would fix the issue

    What We Learned: - Schema is documentation only - LLM behavior is hardcoded in prompts - No validation enforces schema compliance

    Takeaway: Documentation and implementation must be kept in sync manually

    2. Multiple Sources of Truth = Drift¶

    Problem Files: 1. master_parameter_schema.json → "current_output" 2. parameter_extractor.py → "current_rating" 3. category_features_llm.json → "Current Output" 4. product_search.py → ["current_output", ...]

    Solution: Consolidate into single source, generate everything else

    3. Filter Pipeline Order Matters¶

    Issue: 

    1
2
3
    Lucene Search → Score Threshold → Operator Filter
                      ↑
                Eliminates valid products before constraints checked

    Better: 

    1
2
3
    Lucene Search → Operator Filter → Score Threshold
                      ↑
                Constraints applied first, then ranking

    What Helped: - Isolated test script (test_max_300a_query.py) - Server log analysis - Database query validation

    What Would Help More: - Unit tests for constraint matching - Integration tests for full pipeline - Regression tests for operator support
