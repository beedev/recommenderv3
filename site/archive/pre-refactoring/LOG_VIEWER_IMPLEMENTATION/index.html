    Log Viewer Implementation Guide¶

    Overview¶

    A simple, unified log viewing system for ESAB Recommender V2 with: - One API endpoint for all log sources - Simple HTML interface at /static/logviewer.html - Multiple log sources: Application logs, session archives, agent traces, performance metrics - Best practices: Caching, error handling, security, input validation

    üöÄ Quick Start¶

    1. Access the Log Viewer¶

    URL: http://localhost:8000/static/logviewer.html

    2. Select Log Source¶

    Choose from dropdown: - Application Logs - Request/response/error logs from log files - Session Archives - Historical session data from PostgreSQL - Agent Traces - Agent actions, Neo4j queries, LLM extractions - Performance Metrics - Timing and performance data

    3. Apply Filters (Optional)¶

    • Session ID - View logs for specific session
    • Correlation ID - Trace complete request lifecycle
    • User ID - Filter by user (session logs only)
    • Log Level - Error/Warning/Info/Debug (application logs only)
    • Time Range - Start/end datetime
    • Limit - Number of results (50/100/500/1000)

    4. Fetch Logs¶

    Click "Fetch Logs" button or press Enter

    üîå API Endpoint¶

    GET /api/v1/logs¶

    Single unified endpoint for all log sources.

    Query Parameters¶

    Parameter Type Required Description
    source string Yes Log source: application, session, agents, or performance
    session_id string No Filter by session ID (required for agent logs)
    correlation_id string No Filter by correlation ID
    user_id string No Filter by user ID (session logs only)
    level string No Filter by level: debug, info, warning, or error
    status string No Filter by status: completed, abandoned, or finalized
    start_time datetime No Filter logs after this time (ISO8601)
    end_time datetime No Filter logs before this time (ISO8601)
    limit integer No Maximum results (1-1000, default: 100)
    offset integer No Pagination offset (default: 0)

    • Responsive design - Works on desktop, tablet, mobile

    • Fetch Logs - Retrieve logs with current filters
    • Clear Filters - Reset all filters to defaults

    • Enter - Fetch logs (when focused on input/select)

    • Filters saved to localStorage on page unload
    • Restored automatically on page load
    • Persists: source, level, limit

    üîí Security & Best Practices¶

    Security Features¶

    1. Input Validation - Pydantic models validate all parameters
    2. Path Sanitization - Prevents directory traversal attacks
    3. File Size Limits - Max 100MB log files (configurable)
    4. Sensitive Data Redaction - Removes passwords, API keys, tokens
    5. Query Timeouts - 5-second timeout on database queries
    6. Max Request Limits - Limit 1-1000 to prevent abuse

    Performance Optimizations¶

    1. Redis Caching - 5-minute TTL for frequently accessed logs
    2. Stream Parsing - Large files parsed in chunks (memory efficient)
    3. Database Indexes - Indexed on session_id, correlation_id, timestamp
    4. Pagination - Offset/limit to avoid loading all logs
    5. Connection Pooling - SQLAlchemy connection pooling for PostgreSQL

    Error Handling¶

    1. Comprehensive try-catch - All operations wrapped in error handlers
    2. Structured logging - Log all operations for debugging
    3. User-friendly errors - Clear error messages in responses
    4. HTTP status codes - Proper codes (400, 404, 500, 422)
    5. Fallback behavior - Continues without Redis if unavailable

    üìä Log Sources Explained¶

    1. Application Logs¶

    Source: Log files (esab-recommender.log)

    Contains: - HTTP requests/responses - Request timing (duration_ms) - Correlation IDs - Session IDs - Error messages with tracebacks - Middleware logs - Service logs

    Format: - Production: JSON (one event per line) - Development: Structured text with colors

    Filters: level, session_id, correlation_id, time range

    2. Session Archives¶

    Source: PostgreSQL archived_sessions table

    Contains: - Session metadata (created_at, completed_at, duration) - Final state (finalized/abandoned/completed) - Master parameters (user requirements) - Response JSON (selected components) - Conversation messages - Total messages, errors

    Filters: session_id, user_id, status, time range

    3. Agent Traces¶

    Source: PostgreSQL archived_sessions table (JSONB fields)

    Contains: - Agent actions - Agent execution steps - Neo4j queries - Database queries with timing - LLM extractions - Parameter extraction results - State transitions - State machine transitions

    Filters: session_id (required)

    Note: Requires archived session to exist

    4. Performance Metrics¶

    Source: Application logs (extracted timing data)

    Contains: - Request duration (duration_ms) - Endpoint paths - HTTP methods - Status codes - Timestamps

    Filters: time range

    Use Case: Identify slow requests, performance bottlenecks

    üêõ Troubleshooting¶

    Log Viewer Not Loading¶

    Check: 1. Backend running: curl http://localhost:8000/health 2. Static files mounted: Check logs for "Static files mounted from:" 3. Browser console for JavaScript errors

    Fix: Restart backend, clear browser cache

    No Application Logs Showing¶

    Check: 1. Log file exists: ls /home/azureuser/.../logs/esab-recommender.log 2. Permissions: ls -la /home/.../logs/ 3. File size: du -h /home/.../logs/esab-recommender.log

    Fix: - Update LOG_FILE_PATH in .env - Check file permissions (must be readable)

    No Session Logs Showing¶

    Check: 1. PostgreSQL running: curl http://localhost:8000/health 2. Sessions archived: psql -d pconfig -c "SELECT COUNT(*) FROM archived_sessions;"

    Fix: - Archive sessions: POST /api/v1/configurator/archive/{session_id} - Check PostgreSQL connection in .env

    Agent Logs Return "session_id required"¶

    Cause: Agent logs require session_id parameter

    Fix: Enter Session ID in UI or add ?session_id=xyz to API call

    Cache Not Working¶

    Check: 1. Redis running: redis-cli ping 2. Redis connection: curl http://localhost:8000/api/v1/logs/health

    Fix: - Start Redis - Check REDIS_URL or REDIS_HOST in .env - Viewer works without Redis (just slower)

    422 Validation Error¶

    Cause: Invalid parameter values

    Examples: - Invalid source: Use application, session, agents, or performance - Invalid level: Use debug, info, warning, or error - Invalid limit: Must be 1-1000 - Invalid offset: Must be >= 0

    Fix: Check API documentation for valid values
