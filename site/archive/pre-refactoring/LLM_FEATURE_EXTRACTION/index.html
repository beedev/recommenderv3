    LLM-Powered Feature Extraction¶

    Overview¶

    The LLM-powered feature extraction system uses GPT-4 to intelligently analyze product descriptions and extract structured technical specifications, features, and capabilities. This is significantly more powerful than simple text parsing.

    Why Use LLM Extraction?¶

    Traditional Parsing (extract_features_v2.py)¶

    ❌ Only extracts comma-separated attributes from attributes_ruleset ❌ Cannot understand context or infer specifications ❌ No intelligence in categorizing features ❌ Cannot extract ranges or numeric specifications ❌ Results: "MMA", "Inverter", "300A" (raw text)

    LLM-Powered Extraction (extract_features_llm.py)¶

    ✅ Analyzes full product descriptions with context ✅ Infers numeric ranges from text (e.g., "200A - 600A") ✅ Intelligently categorizes features (numeric vs categorical) ✅ Extracts capabilities and processes ✅ Normalizes and structures data ✅ Results: Structured JSON with ranges, categories, and semantics

    cd src/backend
python docs/scripts/extract_features_llm.py

    You'll be prompted to: 1. Select which categories to process 2. Review extracted features 3. Approve/save results

    Automated Demo¶

    1
2
    cd src/backend
python demo_llm_extraction.py

    This runs a quick demo on Powersource category to show capabilities.

    Options¶

    When running the interactive script, you have 4 options:

    Option 1: All categories (recommended after reviewing demo) - Processes all 14 categories - Takes ~5-10 minutes - Uses ~50-100 API calls - Cost: ~$0.50-1.00 in OpenAI credits

    Option 2: Primary categories only - Powersource, Feeder, Cooler, Torch, Interconn - Takes ~2-3 minutes - Focus on main product types

    Option 3: Single category - Test with one category - Useful for iterating on prompts - Fast and cheap

    Option 4: Cancel - Exit without processing

    # In ParameterExtractor
def build_prompt(self, user_message: str, state: str) -> str:
    """Build prompt with feature context"""

    features = self.load_features_for_state(state)

    # Add numeric specs
    specs_text = ""
    for spec in features['numeric_specs']:
        specs_text += f"- {spec['name']}: {spec['display']}\n"

    # Add categorical features
    options_text = ""
    for feat in features['categorical_features']:
        options_text += f"- {feat['name']}: {feat['display']}\n"

    return f"""
Available specifications:
{specs_text}

Available options:
{options_text}

User message: {user_message}

Extract parameters...
"""

    Advantages¶

    1. Intelligent Range Detection¶

    Extracts "300A - 600A" from "300 Ampères to 600 Ampères" automatically

    2. Context Understanding¶

    Understands that "MIG", "TIG", "MMA" are welding processes, not just random keywords

    3. Semantic Grouping¶

    Groups related features: - "Portable", "Heavy duty", "IP44" → Design Features - "MIG", "TIG", "MMA" → Supported Processes - "Air", "Water", "Integrated" → Cooling Options

    4. Normalization¶

    Converts variations to standard format: - "MIG/MAG" → "MIG/GMAW" - "Live TIG" → "TIG/GTAW" - "SMAW" → "MMA/SMAW"

    5. Flexibility¶

    Can extract features from: - description_catalogue (long descriptions) - clean_description (formatted text) - description_ruleset (rule-based text) - attributes_ruleset (comma-separated)

    Cost Considerations¶

    API Usage¶

    • Model: GPT-4 (gpt-4-turbo)
    • Tokens per category: ~2,000-4,000 tokens
    • Cost per category: ~$0.08-0.15
    • Total for 14 categories: ~$1.00-2.00

    Optimization¶

    • Analyzes first 10 products per category (sufficient for pattern recognition)
    • Uses temperature=0.1 for consistency
    • Caches results in JSON file (run once, use many times)

    Limitations¶

    1. API Dependency¶

    Requires OpenAI API key and internet connection

    2. Processing Time¶

    Takes 5-10 minutes for all categories (vs. instant for simple parsing)

    3. Cost¶

    Small cost per extraction (~$1-2), but one-time per database update

    4. Non-Deterministic¶

    May produce slightly different results on re-run (use saved JSON for consistency)

    Best Practices¶

    1. Review Before Deployment¶

    Always review LLM-extracted features before using in production

    2. Validate Ranges¶

    Check that numeric ranges make sense for the category

    3. Update Periodically¶

    Re-run extraction when: - New products are added to database - Product descriptions are updated - You want to refine the LLM prompt

    4. Combine Approaches¶

    Troubleshooting¶

    Issue: API Key Error¶

    1
    Error: OpenAI API key not found
    Solution: Set OPENAI_API_KEY in .env file

    Issue: Rate Limit¶

    1
    Error: Rate limit exceeded
    Solution: Wait a minute and retry, or process fewer categories at once

    Issue: Empty Results¶

    1
    Extracted 0 numeric specs, 0 features
    Solution: - Check that products have descriptions - Review LLM prompt - may need adjustment - Increase number of products analyzed per category

    Issue: Incorrect Ranges¶

    1
    "min": 0, "max": 0
    Solution: - LLM couldn't find specs in descriptions - May need to add more examples in prompt - Check product data quality

    Summary¶

    The LLM-powered approach provides: ✅ Intelligence: Understands context and semantics ✅ Structure: Organized, categorized features ✅ Ranges: Numeric specifications with min/max ✅ Flexibility: Works with any text description ✅ Quality: Professional, consistent output

    Trade-offs: ⚠️ Cost: ~$1-2 per full extraction ⚠️ Time: 5-10 minutes vs. instant ⚠️ Dependency: Requires OpenAI API

    Recommendation: Use LLM extraction for initial feature discovery, then maintain the generated JSON file manually as your "feature catalog."
