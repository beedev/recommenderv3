    Search Method Usage Summary¶

    Most Called Methods (by orchestrator)¶

    1. search_*_smart methods - 30+ calls (main search path)
    2. search_feeder_accessories_smart - 4 calls (regular + preview + proactive + pagination)
    3. search_powersource_accessories_smart - 4 calls (regular + preview + proactive + pagination)
    4. search_interconnector - 4 calls (compound + proactive + mapping + pagination)
    5. _search_by_gin_direct - 2 calls (PowerSource GIN detection)

    Search Method Categories by Usage¶

    • Active Smart Methods: PowerSource, Feeder, Cooler, Torch, PowerSourceAccessories, FeederAccessories
    • Active Traditional Methods: Interconnector, Remotes, Connectivity, FeederWears, InterconnAccessories, FeederConditionalAccessories, RemoteAccessories, RemoteConditionalAccessories
    • Legacy Methods: search_accessories (being phased out)
    • Unused Methods: search_remote_smart, search_interconnector_smart (have traditional counterparts in use)

    Performance Characteristics¶

    Fast Methods (< 50ms)¶

    • _search_by_gin_direct() - Direct primary key lookup
    • _search_by_exact_model() - In-memory fuzzy match

    Medium Methods (50-200ms)¶

    • Traditional Neo4j searches with simple compatibility
    • Lucene searches with small result sets

    Slower Methods (200ms+)¶

    • Triple compatibility checks (Interconnector)
    • Lucene searches with large result sets
    • Searches with complex post-filtering

    Optimization Tips¶

    1. GIN search is fastest → encourage GIN usage
    2. Exact model name is second fastest → cache product names
    3. Smart methods balance speed and flexibility
    4. Lucene is faster than traditional for text queries
    5. Traditional is faster for structured queries

    Inconsistencies and Issues¶

    ❌ Inconsistent Default Limit Values¶

    Problem: Methods have inconsistent default limit values across the codebase.

    Current State: - limit: int = 10 - Used by: search_power_source, all *_smart methods, search_accessories, search_remotes, search_connectivity, search_interconn_accessories, search_feeder_wears - limit: int = 3 - Used by: search_feeder, search_cooler, search_torch, search_interconnector (traditional methods) - limit: int = 4 - Used by: conditional accessory methods (search_powersource_conditional_accessories, search_feeder_conditional_accessories, search_remote_accessories, search_remote_conditional_accessories)

    Impact: - Confusing for developers - same type of operation returns different numbers of results - Orchestrator now overrides with class constants (DEFAULT_SEARCH_LIMIT = 10, PROACTIVE_SEARCH_LIMIT = 10), but default values in method signatures are misleading

    Recommendation: Standardize all to limit: int = 10 for consistency

    ❌ Orphaned Methods (Superseded but Not Removed)¶

    Problem: Traditional search methods exist but are superseded by _smart wrappers, creating confusion about which to use.

    Superseded Methods (Not Called in Orchestrator): 1. search_feeder() → Superseded by search_feeder_smart() 2. search_cooler() → Superseded by search_cooler_smart() 3. search_torch() → Superseded by search_torch_smart() 4. search_powersource_accessories() → Superseded by search_powersource_accessories_smart() 5. search_feeder_accessories() → Superseded by search_feeder_accessories_smart()

    Why They Exist: _smart methods were added later to support Lucene full-text search, but traditional methods were kept as fallback implementation.

    Impact: - Code duplication (traditional logic copied in _smart methods) - Potential for divergence if one is updated and the other isn't - 500+ lines of code that could be removed

    Recommendation: - Option 1: Remove traditional methods entirely, make _smart the only implementation - Option 2: Keep traditional methods as private _search_feeder_traditional() for internal use by _smart
