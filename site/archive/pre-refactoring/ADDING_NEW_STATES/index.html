    Adding New States to the Dynamic State Machine¶

    Complete Guide for Extending S1‚ÜíSN Flow

    This guide explains how to add new states (S8, S9, etc.) to the dynamic state machine without writing any Python code.

    Table of Contents¶

    1. Overview
    2. Prerequisites
    3. Quick Start: Adding a New State
    4. Configuration Files Reference
    5. Step-by-Step Tutorial
    6. Advanced Scenarios
    7. Testing Your Changes
    8. Troubleshooting

    Overview¶

    The dynamic state machine is fully configuration-driven. Adding a new state requires only JSON configuration changes:

    What Changes Automatically¶

    ‚úÖ ConfiguratorState enum (e.g., NEW_STATE_SELECTION) ‚úÖ State transition logic ‚úÖ Processor assignment (single/multi/custom) ‚úÖ Message generation ‚úÖ API responses ‚úÖ State validation

    What You Need to Update¶

    üìù component_types.json - Define the new component üìù state_prompts.json - Add user-facing prompts üìù component_applicability.json - Add applicability rules (if needed)

    Zero Code Changes Required¶

    No Python code changes needed for standard single/multi selection states!

    Prerequisites¶

    Required Knowledge: - Basic JSON syntax - Understanding of S1‚ÜíSN configuration-driven flow - Component categories (PowerSource, Feeder, etc.)

    Tools: - Text editor with JSON support - Access to configuration files in src/backend/app/config/

    Optional but Recommended: - JSON schema validator - API testing tool (Postman, curl, etc.)

    Configuration Files Reference¶

    1. component_types.json¶

    Location: src/backend/app/config/component_types.json

    Purpose: Defines all components and their state machine behavior

    Key Fields:

    Field Type Required Description
    class_name string ‚úÖ Neo4j node label (e.g., "Cable")
    display_name string ‚úÖ User-friendly name
    api_key string ‚úÖ Key in ResponseJSON (e.g., "Cable")
    db_key string ‚úÖ Database identifier
    state_name string ‚úÖ State machine identifier (snake_case)
    state_order integer ‚úÖ Position in sequence (1-based)
    is_mandatory boolean ‚úÖ Must complete? (true/false)
    can_skip boolean ‚úÖ Allow skip? (true/false)
    selection_type string ‚úÖ "single", "multi", or "custom"
    supports_search boolean ‚úÖ Enable product search?
    requires_compatibility_check boolean ‚úÖ Validate compatibility?
    compatible_with array ‚ùå List of required components

    Selection Types:

    • single: Select one product (Feeder, Cooler, Torch, etc.)
    • Uses SingleSelectionProcessor
    • User selects by index or product name

    • Can skip if can_skip: true

    • multi: Select multiple products (Accessories)

    • Uses MultiSelectionProcessor
    • User can select multiple times

    • Type "done" to finish

    • custom: Custom logic required

    • Requires custom processor class
    • Must implement BaseStateProcessor
    • Specify processor_class field

    Best Practices¶

    1. Naming Conventions¶

    State Names (snake_case): - Good: cable_selection, safety_equipment_selection - Bad: CableSelection, cable-selection, select_cable

    API Keys (PascalCase): - Good: Cable, SafetyEquipment - Bad: cable, safety_equipment, CABLE

    Class Names (PascalCase): - Good: Cable, SafetyEquipment - Bad: cable, Cables, safety_equipment

    2. State Order Best Practices¶

    • Keep related states together
    • Put optional states after mandatory ones
    • Keep accessories near the end
    • Finalize is always last
    • No gaps in state_order numbers

    3. Applicability Rules¶

    • Default to "Y" unless component is truly incompatible
    • Use "N" sparingly
    • Document why a component is "N" for specific power sources
    • Update ALL power source entries when adding new components

    4. Prompt Writing Tips¶

    • Be clear and concise
    • Always mention "skip" option if can_skip = true
    • For multi-selection: mention "done" command
    • Use emojis consistently (one per state)
    • Include examples in guidance_bullets
    • Test prompts with non-technical users

    5. Selection Type Guidelines¶

    Use single when: - User selects exactly one product - Product is physical hardware (torch, cooler, etc.) - Selection replaces any previous choice

    Use multi when: - User can select multiple products - Products are additive (accessories, cables, etc.) - "Done" command makes sense

    Use custom when: - Complex validation required - Special business logic needed - Standard processors insufficient
